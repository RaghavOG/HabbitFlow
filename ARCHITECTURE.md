# HabitFlow Architecture

This document explains how HabitFlow stores habit data, computes streaks/analytics, aggregates data for the dashboard, and generates AI insights.

## Data storage model

HabitFlow uses three main MongoDB collections (via Mongoose):

### `User`
User authentication identity (email + password hash) and profile fields.

### `Habit` (metadata)
Represents the habit itself:
- `userId` (ownership)
- `name`
- `color`
- `goalPerMonth`

### `HabitLog` (time-series)
Represents one tracked day (time-series) for one habit:
- `habitId` (reference back to `Habit`)
- `date` (normalized to **UTC midnight**)
- `status` (`true` = done, `false` = missed/not done)

Important constraint:
- Unique index on `(habitId + date)` so **there is at most one log per habit per day**.

This design keeps habit metadata (`Habit`) separate from time-series logs (`HabitLog`), so dashboard queries can efficiently compute stats from log history.

## Streak calculation

Streak is computed from **completed logs only**:

1. Query completed logs (`status: true`) for the habit up to (and including) today.
2. Sort completed dates by `date` descending.
3. Start with `expected = today` and iterate through completed dates:
   - If the next completed date matches `expected`, increment streak and decrement `expected` by 1 day.
   - If not, stop.

This is implemented in `HabitLog.calculateStreak(...)`, using `calculateStreakFromCompletedDatesDesc(...)`.

## Dashboard aggregation (single API call)

The dashboard is served by `GET /api/dashboard`.

High-level flow:
1. Fetch all habits for the authenticated user (`HabitModel.getAllHabits`).
2. Compute the month boundaries in UTC (`getMonthRangeUTC(year, month)`).
3. Fetch logs for **all habits in one query**:
   - `habitId IN [all habitIds]`
   - `date` is within the requested month range
4. Build an in-memory map:
   - `logs[habitId][YYYY-MM-DD] = status`
5. Compute per-habit stats from month logs:
   - `completedDays` = count of `status: true`
   - `successRate = (completedDays / totalTrackedDays) * 100`
   - `progress = (completedDays / goalPerMonth) * 100`
6. Compute `streak` for each habit using the streak helper described above.

The response includes everything the UI needs (habit metadata + per-day status map + aggregated metrics), so the frontend can render the matrix without making many API calls.

## AI insights flow

AI weekly insights are generated by:
- `POST /api/ai/weekly-report`
- `POST /api/ai/report` (alias)

Flow:
1. Backend computes weekly “facts” from MongoDB (habit list + logs in the week range).
2. The facts are either:
   - sent to OpenAI (if `OPENAI_API_KEY` is set), or
   - handled by deterministic fallback logic (if the key is missing).
3. The result is stored for caching in `AIWeeklyReport`:
   - unique index on `(userId + weekKey)`
4. The API returns the generated bullet points to the frontend.

The dashboard displays these bullets in the **AI Insights** card.

## PWA architecture

HabitFlow uses Next.js and includes a `next-pwa` dependency, intended for Progressive Web App support.

Recommended PWA architecture:
- Cache the Next.js “app shell” (static assets) so the dashboard loads instantly.
- Cache the most recent dashboard data (e.g. latest month response) so the matrix remains readable offline.
- Use background sync to refresh logs when connectivity returns.
- Continue using `redux-persist` for client-side state persistence across navigation.

Note: the service worker/runtime wiring may be added in a follow-up change set; this document describes the intended architecture for PWA support.

## Electron architecture

HabitFlow is designed to be packaged as a desktop app (Windows .exe).

Typical Electron architecture in this project:
- Electron main process wraps the Next.js UI (dev server or production build).
- The UI continues to talk to the backend through the existing Next.js API routes.
- Auth remains cookie-based (httpOnly JWT cookie), so the web layer keeps ownership/security logic centralized.

The repo includes Electron dependencies (and the project structure anticipates an Electron folder), but desktop main-process wiring is expected to be handled in the packaging layer.
